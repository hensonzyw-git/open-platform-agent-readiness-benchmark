## Interpretation (matters more than the number)

- **Why 100%**: the signing doc ships full reference implementations in 5 languages. When a doc ships working code, doc-only FPSR is near-ceiling — this is a *strong* Layer-1/Layer-2 signal for the signing endpoint, but a *low-information* task for ranking platforms.
- **Real findings surfaced by the exercise** (more valuable than the 100%):
  1. **Doc site is a JS-SPA** — `WebFetch`/static GET returns an empty shell. Layer 4「可 fetch 文档」fails: an agent limited to HTTP GET cannot even read the docs.
  2. **Worked example is internally inconsistent** — input JSON `timestamp=1603354338917` vs sign-string `1603353500369`. Didn't bite here (clean vector) but is a latent Semantic Clarity defect.
  3. **Latent unverifiable trap**: for array-of-nested-objects, the Java reference preserves key order while the Python reference sorts nested keys — the doc contradicts itself, AND the correct interpretation can't be confirmed without a live sandbox call. This is exactly where doc-only reaches its limit.
