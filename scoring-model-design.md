# Agent-Ready：开放平台 Agent 友好化评分模型

> 一句话定位：**一个行为性 benchmark，量一个开放平台能不能让 AI agent 把整个集成应用做出来。**
> 测量单位是**端到端场景**：把平台文档 URL + 一句开发者需求扔给 agent，看它能不能自主产出**全链路**的 API 调用流程与代码。两条正交头号轴：
> - **轴 A 成功率** Task Success Rate（TSR）—— agent 不靠人工干预、在预算内做对的比例；
> - **轴 B 成本** tokens-to-success —— 做对花了多少 token。
> 成功与否不再是 0/1，而是按"恢复需要多重的人工干预"分成 **A0–A4 自主度阶梯**。

---

## 0. 这份文档要解决什么

把"开放平台 agent 友不友好"这个**模糊判断**，变成一个**可测量、可复现、可驱动决策**的评分模型。产出四样东西的设计：

1. **测什么**（端到端场景集，按集成模式跨平台可比）；
2. **怎么测**（两条头号轴 + 自主度阶梯 + doc-only/live 两种验证深度）；
3. **怎么打分 + 怎么呈现**（双层架构 + scorecard + 榜单）；
4. **场景四件套**作为冻结交付物（这是 benchmark 的本体，不是配料）。

两个用途**等权**服务，共用同一套评分：
- **公开 benchmark**：横评 Stripe / Shopify / 微信 / 飞书，发 GitHub + 方法论文章；
- **POP 改造前后**：用同一把尺子量自家改造的效果增量。

---

## 1. 第一性原理：到底在评什么

**被测对象不是"能不能调通一个 API"，是"把 URL 扔过去，整个应用能不能按开发者需求做出来"。**

退回到最根本的判断：一个 agent，拿到**平台文档 URL + 开发者需求**，**不靠人工干预**就产出**全链路正确集成**（选对接口、字段在多步之间正确流转、单位/幂等/错误处理都对）的概率有多高。这个 outcome 就是平台 agent 友好度的硬定义。

由此推出四条设计约束：

1. **只测端到端，不测单接口。** 单看一个 API doc-only 能不能调通**毫无意义**——文档稍微清楚点、附了参考代码，前沿模型必然贴顶（已实测：POP 签名、寄售单接口都 100%，什么信息都没给）。真正的难点——"116 个 API 里选哪个""skuId 从哪步来""price 单位是不是分"——只在**多步链路**里才暴露。

2. **结果优先于过程，但成功不是 0/1。** 头号指标必须**行为性**（agent 真去做、做没做对），这是硬验证器、难注水。但"做没做对"要按**恢复成本**细分：有的失败给个提示就能绕过去，有的喂了代码也救不活、必须人亲手写——这两种平台友好度天差地别，分数必须不同（见 §3.2 自主度阶梯）。

3. **诊断层解释结果。** 光有 TSR 知道"差"，不知道"为什么差、改哪里"。四层框架（API Quality → Semantic Clarity → Navigation → Callable Interface）当**诊断层**。妙处：**让 agent 恢复的那个提示是哪一种，正好指向是哪一层坏了**——打分与诊断是同一个动作（§3.2）。

4. **可复现是公信力的唯一来源。** 场景集、golden 轨迹、提示阶梯、跑分日志、harness 全部公开冻结，多次跑取均值+方差，别人能复现才算数。

> **心智模型沿用框架原文**：agent 不是"更聪明的用户"，是"有限上下文的代码生成器"。它只能消费被显式表达的语义，不靠经验推断。整套评分都站在这个假设上。

## 2. 评分架构

```
┌─────────────────────────────────────────────────────────┐
│  Tier 1 — 结果层：两条正交头号轴（行为、客观、硬验证器）  │
│  轴 A  成功率 TSR        端到端自主成功率（只数 A4）      │
│         └ 底下铺 A0–A4 自主度阶梯直方图（严重度剖面）     │
│  轴 B  成本 tokens/成功 + iterations/成功                 │
│  二维 scorecard，不折叠成一个分                           │
└─────────────────────────────────────────────────────────┘
              ↑ 解释 / 定位失败原因 ↓
┌─────────────────────────────────────────────────────────┐
│  Tier 2 — 诊断层：四层 rubric（客观项 + 主观项混合）      │
│  Layer1 API Quality / Layer2 Semantic / Layer3 Nav /     │
│  Layer4 Callable Interface                                │
│  阶梯的 rung 直接映射到这四层 → 自带改造 backlog          │
└─────────────────────────────────────────────────────────┘
```

**关键原则一：两层不混成一个分去排名。** 榜单**按 Tier 1 两条轴排序**；四层 rubric 以**雷达图/分项**呈现。可选综合指数（ARI，见 §6）明确标注为诊断辅助，非排序依据——否则退化成可注水的复合分。

**关键原则二：成功率与成本正交，分两条轴报，不折叠。** 一个平台可以 TSR 高却 token 成本极高（典型：文档站 JS-SPA，agent 只能靠截图/OCR 绕，最终做对但代价 10×）。只看成功率会给它满分；成本轴才让它露馅。**成本轴还是 Layer 4「可 fetch 文档」改造价值的唯一可见度量**——该改造往往不动成功率（agent 靠截图也能成），却把 token 砍掉一个数量级。

**关键原则三：成功不是 0/1，是阶梯。** 走不通不直接记 0，按"恢复需要多重干预"分 A0–A4（§3.2）。头号 TSR 只数完全自主（A4，最硬最难注水）；阶梯直方图作为严重度剖面，区分"点破就行"和"喂码也没用"。

## 3. Tier 1：两条头号轴 + 自主度阶梯

### 3.1 指标层级

| 指标 | 轴 | 定义 | 用途 |
|---|---|---|---|
| **TSR（unaided）** | A 成功率（头号） | 端到端场景里 agent **不靠任何提示**走通全链路（A4）的比例，对场景集取均值 | 榜单排序、对外引用 |
| **自主度阶梯剖面** | A 成功率（头号伴随） | 每场景落在 A0–A4 哪一级的分布直方图 | 严重度区分、改造 backlog |
| **Tokens-to-success** | B 成本（头号） | A4 成功 run 完成全链路消耗的总 token（input+output+图像）均值 | 榜单第二轴、量 Navigation/Callable |
| **Iterations-to-success** | B 成本（头号，live） | live 轨道里跑到 green 经历的 runtime 报错—修复回合数 | "first-pass 性"落到这里：要不要靠试错才成 |
| **Criterion pass rate** | 诊断 | 链路内步骤级验收断言的通过率 | 失败定位、改造前后细粒度对比 |
| **Tool-call count** | 诊断 | 完成任务的平均工具调用次数 | 辅助诊断；**会低估 L4 惩罚**——截图是 1 次调用却吃几千 token |

> **命名说明（v0.3 改动）**：旧"FPSR / First-Pass Success Rate"退役。原因：它字面说"第一次尝试"，定义却是"不靠人工修改"——名实不符；且 agent 在端到端里必然循环自纠，"第一次"几无意义。新头号轴叫 **Task Success Rate（TSR）**，钉死为「不靠人工干预、在 token 预算内，产出满足全链路全部验收断言的集成」。doc-only 单次采样那一档可形式化标注「≈ pass@1」。"是否靠试错才成"这层意思，挪进成本族的 **iterations-to-success**。

### 3.2 自主度阶梯（A0–A4）：核心打分机制

走不通不记 0。卡住后，harness 按**预先冻结**的阶梯**逐级注入提示**，重跑，记录它在哪一级恢复。**提示是哪一种，正好暴露是哪一层坏了——打分即诊断。**

| 等级 | 卡住后靠什么救活 | 指向层 / 对应改造 | 通俗 |
|---|---|---|---|
| **A4 自主** | 不用救，自己走通全链路 | — | 理想 |
| **A3 指路可救** | 给导航指针（"文档在此 URL / 有 Fact Sheet"） | L3/L4 导航·可 fetch | 人介入就能绕过去 |
| **A2 点破可救** | 给语义提示（"price 单位是分""用 skuId"） | L2 语义·Integration Notes | 人介入就能绕过去 |
| **A1 喂码才救** | 必须把字段流转/代码直接递给它，平台表面推不出来 | L1·文档缺失 | 必须人亲手写 |
| **A0 死锁** | 递了代码也跑不通；能力缺失/文档自相矛盾 | 能力层硬伤 | 人亲手写也没用 |

打分与呈现：
- **头号 TSR = A4 占比**（最硬、不可注水）。
- **阶梯直方图** = A0–A4 分布，作严重度剖面：一个"卡在 A2、点破就行"的平台，剖面上明显好过"掉到 A0、喂码也救不活"的平台——**这就是"别一刀切 0"的实现**。
- 想要一个标量（如平均自主度 0–4）只能当**诊断辅助**，不进排序（防可注水复合分）。
- **阶梯天然产出改造 backlog**：大量卡 A2 → 补 Integration Notes；卡 A3 → 补 Fact Sheet / 可 fetch 文档；掉 A0/A1 → API 能力或文档硬伤，要动接口。

> **提示阶梯必须事先冻结**（每场景×每平台预写 A3/A2/A1 的具体提示）。否则会"钓"一个刚好让它过的提示——这是"假绿"的变种。

### 3.3 doc-only 与 live：同一个任务，两种验证深度

旧版"doc-only 测单接口、live 测执行、两者粒度不同"的别扭已消除。现在它俩是**同一个端到端任务**，只差最后一步怎么判：

```
同一端到端场景：URL + 开发者需求 → agent 产出全链路流程 + 代码
   ├─ doc-only：拿 golden 轨迹静态比对——接口选对没、字段链通没通、
   │            单位/幂等/错误处理对没对。不执行。
   │            零账号，五平台完全一致，任何人可复现 → 公开榜单唯一可比指标。
   └─ live：    同一条链，真打沙箱，确认全链路跑通。仅 POP。
```

| | doc-only TSR | live TSR |
|---|---|---|
| 判成功 | 全链路轨迹匹配 golden（≈ 静态 pass@1） | 沙箱真执行成功 |
| 覆盖平台 | 5 平台全做（含 POP） | 仅 POP |
| 抓得到 | "agent 把文档暴露的一切都用对了" | 上面 + runtime-only 失败 |
| 抓不到 | runtime-only（签名时序、真实幂等、文档没写的必填字段、服务端怪癖） | — |

**关键招：POP 同时跑两种。** doc-only 是可比基线（和竞品同尺、进公开榜），live 是额外深度（抓 doc-only 够不到那截，如今天发现的签名样例歧义——纯文档无法判定哪个对，必须 live）。两个数应同向变动 → 同向即验证了 doc-only 作为代理的有效性。

### 3.4 跑分控制（让数可信）

| 控制项 | 做法 | 为什么 |
|---|---|---|
| **模型标准化** | 固定 ≥2 个模型（如 Claude + GPT），分别报告 | 避免单模型 artifact；agent 友好度不该只对一个模型成立 |
| **方差控制** | 每场景跑 K 次（建议 3–5），报均值 ± 方差 | 呼应"假绿"——单次跑绿不可信 |
| **上下文隔离** | doc-only 时 agent 只准从该平台公开 surface 起跑，禁联网搜其他来源 | 隔离"文档质量"与"模型外部知识" |
| **Grounding 要求** | agent 对每个字段/接口选择**引用文档出处**，无出处的正确不计为"文档驱动成功" | 见 §7 训练数据污染 |
| **提示阶梯冻结** | A3/A2/A1 提示逐场景×逐平台预注册，跑前冻结 | 防"钓提示"作弊；保证 rung 语义跨平台一致、可比 |
| **Token 预算上限** | 每场景设硬上限（如 200k token），超限判 A0 | 让 tokens-to-success 有界可比；防 agent 在烂文档上空转刷成本 |
| **冻结** | 场景四件套 + harness + 模型版本 + 工具集，跑一个平台前全部冻结并记录 | 可复现 + 防事后调任务美化结果；工具集冻结对成本轴尤其关键 |

## 4. 端到端场景集（benchmark 的本体）

### 4.1 跨平台可比：按"集成模式"定义场景，不按业务对象

五个平台跨三个域：支付（Stripe）、电商（Shopify、POP/得物）、IM/协作（微信、飞书）。"寄售挂单"在飞书不存在——端到端场景比单个能力更难直接对齐。

**解法：场景按"集成模式"（integration pattern）定义，再在每个平台实例化。** 一个集成模式是所有开放平台都有的**多步链路骨架**，例如：

> **模式 P1「依赖式幂等写」**：鉴权 → 读一个实体 → 据其输出派生一个值 → 幂等地写一个依赖实体 → 处理文档列明的错误。

它在各平台落成具体场景（POP=货号→定价→挂单；Stripe=取 customer→建 PaymentIntent；飞书=取用户→发消息建记录…）。TSR 在模式上聚合，于是异构平台可比。

### 4.2 原 archetype 的去向：从"独立任务"降级为"步骤类型 / 诊断标签"

旧 T1–T6（鉴权 / 读 / 带幂等的写 / 分页 / webhook / 错误恢复）**不再是独立打分任务**（单接口测量无意义，§1）。它们保留为两个用途：
1. **步骤类型**——端到端场景由它们**串成链**（鉴权→读→写…），保证不同平台的场景骨架可比；
2. **诊断标签**——链路哪一步断了，用步骤类型标注，直接指向四层里哪一层。

### 4.3 场景"四件套"（冻结交付物）

每个场景 = 平台 × 集成模式，落成一份冻结的四件套：

```
① 开发者需求（用户意图，一句话）
② golden 全链路轨迹（每步用哪个接口、字段从哪步的输出流到哪步的输入）
③ 步骤级验收断言（具体可判定：单位/幂等键/枚举/错误码/字段来源）
④ 提示阶梯（A3 指针 / A2 点破 / A1 代码，逐级预写并冻结）
```

**示例：POP × P1 → Playbook A（货号 → 定价 → 挂单）**

```
① 需求："帮商家把货号 DB3597-100 在美区挂单，按平台参考价定价，处理重复提交"

② golden 轨迹：
   Step1  /intl-commodity/by-article-number   入: articleNumber,region   出: skuId
   Step2  /recommend-bid/price                 入: skuId(来自S1), biddingType, currency  出: globalMinPrice
   Step3  /submit-bid/normal-autonomous-bidding 入: skuId(S1), price(基于S2), quantity, requestId(新UUID)  出: sellerBiddingNo

③ 步骤级验收断言：
   [S1] 用 by-article-number 而非搜索接口；region 必填
   [S3] price 用货币最小单位（分），$154.00 → 15400          ← Semantic 陷阱
   [S3] skuId 取自 S1 输出，且优先于 globalSkuId              ← 字段来源 + Semantic
   [S3] requestId 为新 UUID 幂等键                           ← Semantic 陷阱
   [S3] 处理 DUPLICATE_REQUEST_ID 等文档列明错误码           ← API Quality
   [全]  每个选择附文档出处引用                               ← Grounding

④ 提示阶梯（冻结）：
   A3 指针："挂单流程见 Playbook / submit-bid 文档在 <URL>"
   A2 点破："price 单位是分；skuId 来自 by-article-number；requestId 是幂等键"
   A1 代码：直接给出 S1→S3 的字段流转骨架
```

> 全链路任一断点 → 该场景按在哪级恢复记 A4/A3/A2/A1/A0；步骤级 criterion pass rate 另记（如 5/6）作诊断。

### 4.4 "场景准备本身就是关键测试内容"

判 A0/A1 需要**真的知道"正确答案到底存不存在"**：POP 你清楚；竞品得把文档啃透、必要时上免费沙箱验证 golden（Stripe test mode / Shopify dev store / 微信测试号 / 飞书 free tenant）。**这份啃透 + 四件套撰写，是整个 benchmark 最费脑、最值钱、该被冻结和公开的交付物**，不是配料。

## 5. 四层诊断 Rubric（Tier 2）

阶梯的 rung 指向哪一层（§3.2），这一层给出**该层内部为什么坏**的细分。每个维度标注**客观（O）/ 主观（S）**：客观项走确定性检查，主观项走 LLM-as-judge（带校准）。

### Layer 1 — API Quality（地基，多为客观）
| 维度 | O/S | 打分方式 |
|---|---|---|
| 结构化错误码 | O | 错误响应是否含机读字段（error_code/error_type/hint）；按样本接口比例打分 |
| 幂等性 | O | 写接口是否幂等、幂等键是否显式标注 |
| 参数命名一致性 | O | 同一语义概念是否全平台单一字段名；抽样检测冲突率 |
| Scope 粒度 | S | 是否按场景聚合、能力语义是否自解释（LLM-judge + 人工校准） |

### Layer 2 — Semantic Clarity（对 TSR 影响最直接，混合）
| 维度 | O/S | 打分方式 |
|---|---|---|
| Integration Notes / 陷阱说明 | O+S | 是否存在（O）+ 是否覆盖真实陷阱（S，LLM-judge） |
| 枚举值集中页 | O | 是否有独立 Reference 页、是否散落 |
| Error Handling 示例 | O | 详情页是否配 2–3 个错误场景完整示例 |
| 字段名→语义可推断性 | S | 给字段名+描述，judge 能否推出单位/含义；**用你的人工标注当 ground truth 校准** |
| 示例自洽性 | O | 文档的 worked example 输入能否复现其声称的输出（实测 POP 签名样例不自洽：输入 timestamp 复现不出 claimed sign） |

### Layer 3 — Navigation（解决有限上下文，混合）
| 维度 | O/S | 打分方式 |
|---|---|---|
| 场景化 Playbook | O+S | 是否存在（O）+ 场景内字段流转是否唯一确定（S） |
| Fact Sheet 全量索引 | O | 是否一次 fetch 拿全、是否依赖 JS/多步交互 |
| llms.txt | O | 是否存在 + 内容是否聚焦陷阱/枚举/playbook |
| 工具调用次数 / token 成本 | O | 即轴 B，行为性测得 |

### Layer 4 — Callable Interface（工程最重，多为客观）
| 维度 | O/S | 打分方式 |
|---|---|---|
| 可 fetch 文档 API | O | GET 是否返回机读 markdown/json（实测 POP 文档站为 JS-SPA，静态 GET 返回空壳、参数表需截图）|
| OpenAPI Spec 覆盖率 | O | 是否存在 + 覆盖接口比例 |
| MCP 接口暴露率 | O | 是否提供 MCP + 覆盖能力比例 |

### LLM-as-judge 的防注水设计
主观项不打"感觉分"。每个主观维度：(1) 给 judge **具体锚点示例**（什么算 5 分、什么算 1 分）；(2) 要求 judge **逐条引用证据**再打分；(3) 用**你的人工标注**在抽样集上校准 judge，报告一致性（如 Cohen's κ）。judge 不可靠的维度降级为纯客观项或人工。

## 6. 综合呈现

**榜单（对外）**：二维呈现——主排序 **doc-only TSR（A4 占比）**，并列 **tokens-to-success**（同模型同 harness 下可比）。理想平台落在「高 TSR + 低 token」象限；JS-SPA 这类平台显形为「TSR 尚可 + token 高」。附均值±方差、分模型。

## 7. 效度威胁与解药（这一节是 eval 产品 sense 的核心）

| 威胁 | 说明 | 解药 |
|---|---|---|
| **训练数据污染** | Stripe 在所有模型训练集里，POP 不在。doc-only TSR / 低 token 会偏向 Stripe，**不是因为它文档好，而是模型背过**。 | (1) **Grounding 要求**：无出处的正确不计成功，测文档非记忆；(2) 选不易被记忆的场景/边缘参数；(3) 可选 **doc-ablation**：抹平台名/换代号跑；(4) 端到端多步链路本身**强抗记忆**（POP 特定 5 步流没人背得过）；(5) **POP 改造前后 delta 最干净**（同法测、无记忆优势）。 |
| **harness 泄题（假绿之一）** | **本人实测踩过**：把 required 参数预先写进任务，等于喂答案，TSR 虚高到 100%。 | 任务只给业务意图 + 凭证，**绝不预喂接口/必填字段/单位**；每个场景跑前过一遍**泄题审计**；从 URL 起跑、渐进披露（§3.2）。 |
| **钓提示（假绿之二）** | 事后挑一个刚好让 agent 过的提示，伪造低 rung 恢复。 | 提示阶梯 A3/A2/A1 **预注册冻结**，rung 语义跨平台统一。 |
| **自卖自夸** | 自造标准又恰好证明自家变好。 | (1) POP 同法上同一张榜，无特殊待遇；(2) 改造前预注册四件套，事后不可改；(3) 公开方法论+golden+run log；(4) 第三方平台锚定刻度，POP 的数只在相对位置有意义。 |
| **假绿（单次跑通即宣称成功）** | 呼应本人血的教训。 | 每场景跑 K 次取均值±方差；公开原始 run log；验收断言机器可判定，不靠人眼"看着对"。 |
| **场景集偏置** | 场景选得巧可美化某平台。 | 场景按跨平台中立的"集成模式"定义；四件套冻结+公开+版本化；欢迎外部 PR 加场景。 |

## 8. POP 改造前后协议

```
1. 改造前：冻结场景四件套 / harness / 模型 → 跑 doc-only + live → 记为 baseline
2. 实施改造（已提需的 POP agent 友好化改造点）
3. 改造后：同一冻结配置重跑 → 记为 post
4. 报告（两条轴 + 阶梯都报）：
   - 轴 A：doc-only ΔTSR（可比，进公开榜）+ live ΔTSR（额外深度）
           + 阶梯迁移（多少场景从 A2/A1 升到 A4）
   - 轴 B：Δtokens-to-success——L4「可 fetch 文档」改造的价值主要落这里
           （TSR 可能不动，token 砍一个数量级）
   - 同轴改造前后同向 → 互证；列出每个改造项消除了哪个失败模式 / 成本黑洞
```

证据强度示例：`"POP 改造后 doc-only TSR 从 38% → 71%（其中 9 个场景由 A2『price 单位』升到 A4，因新增 Integration Notes）；tokens-to-success 从 48k → 6k（约 8×，因可 fetch markdown 消除截图/OCR 路径）；live 同向。"` —— 两条正交硬数字 + 阶梯迁移，比单一指标更能体现改造全貌，是简历里"shipped & measured"的开发者侧硬指标。

## 9. v1 范围与执行路线

**v1 锁定：**
- 先 **1 个集成模式（P1 依赖式幂等写）× 5 平台** 跑穿端到端闭环，再扩到 3–4 个模式；
- doc-only 全平台 + POP 额外 live；
- 2 个模型 × 每场景 5 次；
- 两条头号轴（TSR + tokens-to-success）+ A0–A4 阶梯剖面 + 四层雷达 + Top 失败模式 + Top 成本黑洞；ARI 暂缓。

**建议落地顺序（先客观闭环、再扩主观）：**
1. **写第一个场景四件套**（POP × P1 = Playbook A，含提示阶梯）——最费脑、最值钱的设计活；
2. 搭**渐进披露 harness**（从 URL 起跑、卡住注入下一级提示、重跑、按 rung 计分）→ 先跑通 POP doc-only 闭环；
3. 加 Stripe（doc 可静态抓）做第一张横评对照，验证阶梯打分顺不顺手；
4. 接 LLM-as-judge 做诊断层 + 步骤断言校准；
5. 补齐另外三个平台；
6. POP live 轨道 + 改造前后 ΔTSR/Δtoken。

**Surface 分工**：评分模型设计 / 竞品文档研究 / 场景撰写在 Cowork；harness 实现+跑分+GitHub commit 在 Code 标签。

## 10. 开放问题（需你拍板）

1. **集成模式数量**：v1 先 1 个模式跑穿，还是一上来铺 3 个（依赖式幂等写 / 分页对账 / webhook 处理）？
2. **模型选型**：固定哪两个模型当标准被测 agent？（Claude + GPT？要不要加一个国产模型以示中立？）
3. **doc-only 的"公开 surface"边界**：只算官网文档站，还是含官方 SDK README / OpenAPI 文件？边界要写死才可复现。
4. **grounding 强制度**：无出处的正确"不计成功"会不会太严？还是降级为"标记但仍计分 + 单列 grounded-TSR"？
5. **阶梯粒度**：A0–A4 五级够不够？提示是按"整条链卡住"注入，还是按"步骤"注入（更细但更贵）？
6. **tokens-to-success 跨模型可比性**：图像 token 各模型口径不同，是否只在「同模型」内做改造前后/平台对比，公开榜分模型列两套 token？还是引入归一化基准任务做缩放？
7. **沙箱凭证**：POP live 轨道依赖得物沙箱账号是否可得（决定 §8 能不能落地）。
